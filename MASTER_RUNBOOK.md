## 8. GCP IAM / OIDC / GHA Permission Wall (gp-phantomvision-dev) — Root Cause + Exact Fix Commands

**Root Cause (from GH diagnostics subagent — 2026-05-26):**  
All GitHub Actions runs (both `deploy.yml` and `deploy-cloudrun.yml`) hit a hard permission wall after OIDC token exchange. The WIF pool/provider may exist in partial form and the SA may authenticate, but subsequent `gcloud` calls (especially `gcloud services enable`, Artifact Registry push, Cloud Run deploy, and any Cloud Build fallback) fail with 403 "Permission Denied" or "Service account ... does not have ... permission".  

Specific missing grants identified:
- `roles/serviceusage.serviceUsageAdmin` (workflow explicitly calls `gcloud services enable`; Consumer is insufficient for some org policies / first-time enables).
- `roles/cloudbuild.builds.editor` (required when using Cloud Build path in deploy.sh or if triggers/builds are involved; default Cloud Build SA also needs storage on the project's build bucket).
- `roles/storage.admin` (or objectAdmin) **specifically on the Cloud Build bucket** `gs://gp-phantomvision-dev_cloudbuild` (and/or project-level storage perms) — this is the classic hidden blocker for GHA + AR + Cloud Run source uploads and build logs.
- Additional standard roles often missing in practice on this project: full Run/AR + iam.serviceAccountUser (for Cloud Run to use other SAs) + logging + monitoring viewer/editor in some cases.
- Drift between workflows: `deploy.yml` comments reference `github-actions-deployer@...` + generic pool name; `deploy-cloudrun.yml` hardcodes `github-actions@...` + `projects/431115644565/locations/global/workloadIdentityPools/github/providers/github`. This causes secret mismatches and failed auth in GHA.
- No long-lived keys; pure OIDC/WIF path is correct but the IAM binding + grants were never fully applied.

**This wall is the single reason no clean Cloud Run revision has ever been produced by CI.** Local `./deploy.sh` (with your ADC) may succeed where GHA fails. The 4th subagent output + workflow logs + `gcloud projects get-iam-policy` / `gsutil iam get` confirmed the gaps.
